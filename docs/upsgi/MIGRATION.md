# upsgi migration guide

## Who this guide is for
This guide is for operators already running PSGI applications on uWSGI who want
to move to the PSGI-only upsgi fork with minimal risk.

## Migration summary
What stays the same:
- prefork master/worker model
- PSGI app loading with `--psgi`
- `--http-socket` front-door serving
- `--static-map` baseline static serving
- request logging and forwarded-for logging
- Perl helper surface retained in v1

What changes intentionally:
- the default build is PSGI-first
- `--http-modifier*` remains accepted but is runtime-inert
- `--http-socket-modifier*` and `--https-socket-modifier*` remain accepted but
  are runtime-inert
- `--perl-no-die-catch` remains accepted but is runtime-inert
- `--log-exceptions` is the canonical way to turn on explicit PSGI exception
  visibility for debugging

## Safe migration procedure
1. Build or install the upsgi binary from the default profile.
2. Copy an existing working PSGI config.
3. Keep the launch shape the same:
   - `master = true`
   - `workers = N`
   - `http-socket = :PORT`
   - `psgi = app.psgi`
4. Keep compatibility-only flags in place for the first validation pass if that
   reduces migration risk.
5. Start with `need-app = true`, `strict = true`, `vacuum = true`, and
   `die-on-term = true`.
6. Run the fork regression or your local smoke tests against the migrated app.
7. After validation, remove compatibility-only flags from the config so the
   long-term config reflects the actual supported surface.

## Option mapping
| Existing config item | upsgi status | Action |
|---|---|---|
| `http-socket = :9090` | supported baseline | keep |
| `psgi = app.psgi` | supported baseline | keep |
| `static-map = /assets=./public` | supported baseline | keep |
| `log-x-forwarded-for = true` | supported baseline | keep |
| `http-modifier1 = 5` | compatibility-only | may keep for first migration pass; remove later |
| `http-modifier2 = 0` | compatibility-only | may keep for first migration pass; remove later |
| `http-socket-modifier1 = 5` | compatibility-only | may keep for first migration pass; remove later |
| `http-socket-modifier2 = 0` | compatibility-only | may keep for first migration pass; remove later |
| `https-socket-modifier1 = 5` | compatibility-only | may keep for first migration pass; remove later |
| `https-socket-modifier2 = 0` | compatibility-only | may keep for first migration pass; remove later |
| `perl-no-die-catch = true` | compatibility-only | remove after validation |
| `log-exceptions = true` | supported debug behavior | enable only when server-side exception visibility is needed |

## Old-to-new config examples
### Recommended baseline config
```ini
[uwsgi]
master = true
workers = 2
need-app = true
strict = true
vacuum = true
die-on-term = true
http-socket = :9090
psgi = app.psgi
static-map = /assets=./public
logto = ./upsgi.log
log-x-forwarded-for = true
```

### Migration-first config with compatibility flags retained
```ini
[uwsgi]
master = true
workers = 2
need-app = true
strict = true
vacuum = true
die-on-term = true
http-socket = :9090
psgi = app.psgi
static-map = /assets=./public
logto = ./upsgi.log
log-x-forwarded-for = true
http-modifier1 = 5
http-modifier2 = 0
perl-no-die-catch = true
```

### Debug exception visibility
```ini
[uwsgi]
master = true
workers = 1
need-app = true
strict = true
vacuum = true
die-on-term = true
http-socket = :9090
psgi = app.psgi
logto = ./upsgi.log
log-exceptions = true
```

## Validation checklist
- baseline startup succeeds
- app loads successfully with `need-app = true`
- HTTP socket requests succeed
- static-map requests succeed
- missing static files fall through to the PSGI app as expected
- logs show forwarded-for data when enabled
- compatibility-only flags do not break startup
- exception visibility is quiet by default and visible only when
  `log-exceptions = true`

## Known migration risks
- configs that depend on modifier values having real runtime meaning need to be
  reviewed; upsgi accepts those options but does not use them to steer runtime
  behavior
- configs that imply full upstream router or gateway breadth should be reviewed
  against `docs/upsgi/DEFERRED_SCOPE.md`
- release packaging should exclude local test artifacts generated by the fork
  harness
- if the application uses `psgix.informational`, validate that the deployed front-end path is still HTTP/1.1 to the app server where informational responses are expected

## FAQ
### Do I have to remove compatibility-only flags immediately?
No. Keep them for the first migration pass if that helps operational safety.
Remove them after validation so the config matches the supported long-term
surface.

### Is `perl-no-die-catch` still needed?
No. It is accepted only for migration compatibility. Default PSGI exception
catch/logging is already off in the fork.

### How do I turn on exception visibility during debugging?
Use `log-exceptions = true`.
